Проведи анализ кода используя памятку "Критерии оценки качества кода".
Строго следуй формулам.
Исключи субъективную корректировку оценок.
**Важно!** Предупреди, если объем контекста не позволяет провести подробный построчный анализ кода!

# ⚠️ Критерии применимости

Этот промпт эффективен **только** при соблюдении условий ниже. Нарушение любого из них приводит к поверхностному или ошибочному анализу.

## Размер контекста

| Объём кода | Применимость | Почему |
|-----------|-------------|--------|
| **1 система / пакет** (до ~3000 LOC) | ✅ Полная | Всё помещается в контекст, каждый файл читается целиком |
| **2–5 систем** (до ~10 000 LOC) | ⚠️ Частичная | Работает для количественных метрик, маркеры проблем могут быть пропущены |
| **Весь фреймворк** (10 000+ LOC) | ❌ Неприменим | LLM переходит в режим каталогизации вместо анализа. Пропускает `async void`, ложную расширяемость, fat interface — всё, что требует построчного чтения |

**Правило:** если код не помещается в контекст так, чтобы каждый файл был прочитан целиком — разбей на пакеты и анализируй по одному.

## Режим работы

Промпт требует **глубокого чтения**, а не обзора:

- **Каждый публичный интерфейс** — проверить на ISP (число членов, смешение ответственностей)
- **Каждый `as ConcreteType`** — проверить на ложную расширяемость
- **Каждый `async void`** — красный флаг, задокументировать
- **Каждый endpoint-класс** — посчитать поля **включая все базовые классы** (суммирование по иерархии)
- **Каждый override** — проверить, не является ли он пустой заглушкой (маркер раздутого базового класса)

Если LLM начинает описывать структуру папок, считать файлы или составлять таблицы namespace'ов — **анализ не идёт**. Это каталогизация, а не оценка качества.

## Порядок анализа

1. **Сначала интерфейсы и базовые классы** — здесь живут архитектурные проблемы
2. **Затем реализации** — проверка на соответствие контрактам
3. **Затем endpoint-классы** — подсчёт полей, цикломатическая сложность
4. **В конце — количественные метрики** по формулам

Не начинать с подсчёта файлов и LOC. Это создаёт иллюзию прогресса при нулевом анализе.

# 📋 Памятка: Критерии оценки качества кода

## 🎯 Ключевые принципы

1. **Абсолютные метрики бесполезны** — 2652 virtual методов или 77K LOC ничего не говорят о качестве
2. **Оценка требует относительных величин** — соотношения между метриками
3. **Код не показывает скорость разработки** — только эмпирические данные (время на фичу, время на изменение)
4. **Хорошая документация плохой архитектуры ≠ хороший код** — документация объясняет как обойти проблемы, а не почему их нет
5. **При оценке нужно использовать только формулы** - нельзя вносить субъективное мнение в оценку если об этом явно не предупредили

---

## 📊 Система оценки (10-балльная шкала по четырём категориям)

### 1. Документируемость (0-10)

| Критерий | Формула | Баллы |
|----------|---------|-------|
| **1.1** Строки документации / системы | `4 × log₁₀(docs_lines / systems + 1) / log₁₀(1000)` | 0-4 |
| **1.2** Комментарии / объекты | Норма: 1/8 для полей/методов, 1/3 для классов | 0-3 |
| **1.3** Контракты и примеры | `3 × min(1, detailed_docs × completeness / systems)` | 0-3 |

---

### 2. Масштабируемость и гибкость (0-10)

| Критерий | Формула | Баллы |
|----------|---------|-------|
| **2.1** Нейтральные классы / всего (без endpoint) | `1,5 × log₁₀(neutral / (total - endpoint) × 100 + 1) / 2` | 0-2,5 |
| **2.2** Атомарность (среднее полей на endpoint) | `2 × log₁₀(30 / avg_fields)`, норма = 3-5 полей | 0-2 |
| **2.3** Чистота паттернов | `1,5 × clean_patterns / detected_patterns` | 0-1,5 |
| **2.4** Однонаправленность ссылок | `2 × (1 - violations / total_dependencies)` | 0-2 |
| **2.5** Глубина наследования (DIT) | `1 × max(0, 1 - max(0,(avg_depth - 1) / 4))` | 0-1 |
| **2.6** Цикломатическая сложность | `1 × max(0, 1 - max(0,(avg_complexity - 5) / 15))` | 0-1 |

**Определения:**
- **endpoint-класс:** класс без ссылок на другие классы с логикой (чистые модели допустимы), кроме компонентов движка (Button, Text, Image и т.п.). Render классы работающие с GUIStyle, EditorGUI и т.п. не входят в endpoint классы.
- **нейтральный класс:** класс без ссылок на другие классы с логикой, кроме ссылок через интерфейсы или на чистые модели данных. Шины данных выполненные по архитектуре синглтон считаются как 0.5
- **DIT (Depth of Inheritance Tree):** количество уровней наследования от базового класса
- **Цикломатическая сложность:** количество независимых путей в методе (if/else/switch/for/while/catch/?:)

**Важное:**
- При проверке на чистоту паттернов **обязательно** проверять на сброс Lazy-кешированных параметров

**Кол-во полей в классе определять с учетом его предков (суммирование)**

---

### 3. Поддерживаемость (0-10)

| Критерий | Формула | Баллы |
|----------|---------|-------|
| **3.1** Средний размер класса (LOC) | `2 × min(1, max(0, 1 - log₂(avg_lines / 100) / 3))` | 0-2 |
| **3.2** Системы / классы (без интерфейсов и моделей) | Чем больше — тем лучше | 0-2 |
| **3.3** Системы / константы | `min(1.5, 1.5 × max(0,(1.3 - (constants / system)^(1/3)))))` | 0-1,5 |
| **3.4** Документируемость % | `2 × (doc_score / 10)` | 0-1,5 |
| **3.5** Дублирование кода (DRY) | `2 × max(0, 1 - duplicate_blocks / total_methods)` | 0-2 |
| **3.6** Магические значения | `1 × max(0, 1 - magic_numbers / (systems × 10))` | 0-1 |

**Определения:**
- **duplicate_blocks:** блоки ≥5 строк с совпадением ≥80%
- **magic_numbers:** числовые литералы в коде кроме 0, 1, -1, 2 (проверка: `grep -E "[^0-9][3-9][0-9]*[^0-9f]|[^0-9][0-9]{2,}"`)

**Трактовка**
- **не бывает слишком мало констант**. Есть условно-оптимальное кол-во констант на систему. Этот параметр является маркером слишком сильного замусоривания константами.

---

### 4. Производительность архитектуры (0-10)

| Критерий | Формула | Баллы |
|----------|---------|-------|
| **4.1** Аллокации в hot path | `4 × max(0, 1 - new_in_update / update_methods)` | 0-4 |
| **4.2** Кеширование GetComponent | `3 × (cached_getcomponent / total_getcomponent)` | 0-3 |
| **4.3** String операции в циклах | `3 × max(0, 1 - string_concat_in_loops / total_loops)` | 0-3 |

**Определения:**
- **new_in_update:** использование `new` внутри Update/FixedUpdate/LateUpdate/OnGUI
- **cached_getcomponent:** GetComponent вызванный в Awake/Start/OnEnable и сохранённый в поле
- **total_getcomponent:** все вызовы GetComponent в проекте
- **string_concat_in_loops:** конкатенация строк (`+`, `+=`) внутри for/while/foreach

**Быстрая проверка:**
```bash
# Аллокации в Update
grep -n "void Update\|void FixedUpdate\|void LateUpdate" -A 20 | grep "new "

# GetComponent в Update
grep -n "void Update" -A 20 | grep "GetComponent"

# String concat в циклах
grep -n "for\|while\|foreach" -A 10 | grep '\".*+\|+.*\"'
```

## 🚩 Маркеры архитектурных проблем

### Ложная точка расширения
```
Условие: Есть публичный интерфейс IA, реализация A работает 
         с конкретным классом B без интерфейса IB
Признак: `as ConcreteClass` после загрузки "расширяемого" компонента, 
         при этом ConcreteClass обязательно реализует интерфейс IB
Проблема: Подмена верхнего уровня бесполезна без переписывания внутренностей
```

### Смешение Model/View
```
Признак: Интерфейс содержит и Position/Rotation/Scale, и бизнес-данные
Проблема: Нельзя реализовать логику без реализации визуала
```

### Fat Interface
```
Признак: 15+ членов в интерфейсе
Проблема: Нельзя реализовать часть — только всё или ничего
```

### Перегруженность настроек
```
Признак: 20+ полей в Configuration/Metadata классе
Проблема: Невозможно понять взаимосвязи без документации
```

### Низкая Discoverability
```
Признак: Базовая операция требует неочевидного пути
Пример: "Перейти к следующей реплике" → InputManager.GetContinue().AddObjectTrigger(gameObject)
Проблема: Решение нельзя найти через автодополнение или интерфейсы
```

### Ложная точка расширения (детализация)
```
Признак: Класс A (с интерфейсом IA) использует класс B напрямую,
         хотя B имеет интерфейс IB в той же системе

Ключевой вопрос: "Есть ли у B интерфейс в этой системе?"
- Да, есть IB → использование B вместо IB = ПРОБЛЕМА
- Нет интерфейса → прямая зависимость = честная связка (ОК)
- B это endpoint движка (Image, Button) → всегда ОК
```

| Сложность | Оценка | Балл |
|-----------|--------|------|

| 11-20 | Сложный | 0.1-0.4 |
| 20+ | Требует рефакторинга | 0 |

## 🔍 Быстрая проверка (grep-команды)

```bash
# Ложные точки расширения
grep -E "as [A-Z][a-z]+[A-Z]"

# Перегруженность
grep -c "\[SerializeField\]"
grep -c "\[Tooltip\]"

# Fat interfaces  
grep -A 50 "public interface I" | grep -c "{ get\|void \|UniTask "

# Магические строки
grep -c "const string"

# Скрытые зависимости
grep -c "Engine.GetService\|GetServiceOrErr"

# Глубина наследования
grep -E "class \w+ : \w+ " | grep -v "MonoBehaviour\|ScriptableObject"

# Цикломатическая сложность (приблизительно)
grep -c "if\|else\|switch\|case\|for\|while\|foreach\|catch\|\?.*:"

# Аллокации в Update
grep -n "void Update\|void FixedUpdate" -A 30 | grep "new \|Instantiate"

# GetComponent без кеширования
grep -n "void Update\|void FixedUpdate" -A 30 | grep "GetComponent"

# String concat в циклах
grep -B5 -A5 "for\|while\|foreach" | grep '\".*+\|+=.*\"'

# Дублирование (требует jscpd)
npx jscpd --min-lines 5 --threshold 80
```

## 🎯 Итоговая формула качества

```
Влияние 4го раздела считаем более слабым.

Качество = (Документируемость/10 + Масштабируемость/10 + Поддерживаемость/10 + Производительность/20) / 3.5 × 10

Где:
- Документируемость: 0-10 баллов
- Масштабируемость: 0-10 баллов  
- Поддерживаемость: 0-10 баллов
- Производительность: 0-10 баллов

Интерпретация:
- 8-10: Отличная архитектура
- 6-8: Хорошая, с minor проблемами  
- 4-6: Приемлемая, технический долг копится
- 2-4: Проблемная, сложно поддерживать
- 0-2: Критическая, требует переписывания
```

## 📋 Чек-лист для быстрой оценки

### Критические проблемы (любая = красный флаг)
- [ ] Цикломатическая сложность >20 в любом методе
- [ ] DIT >5 уровней
- [ ] Fat interface >15 членов
- [ ] new/Instantiate в Update без пула
- [ ] GetComponent в каждом кадре
- [ ] >30 полей в endpoint классе

### Жёлтые флаги (накопление = проблема)
- [ ] Пустые XML-теги документации
- [ ] Магические числа без констант
- [ ] Дублирование >10%
- [ ] String конкатенация в циклах
- [ ] Статические контроллеры с состоянием
