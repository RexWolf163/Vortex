# Правила оформления документации пакетов

## Стиль

- Академический, сухой, фактологический тон
- Без маркетинговых формулировок
- Без разговорных оборотов
- Краткие формулировки, без повторов

## Структура README

1. **Заголовок** — название системы
2. **Назначение** — зона ответственности системы. Список возможностей. В конце: что вне ответственности
3. **Зависимости** — Core и Unity отдельно
4. **Архитектура** — схема, таблицы, описание паттернов
5. **Контракт** — вход, выход, гарантии, ограничения
6. **Использование** — пошаговый workflow интеграции. Для пакетов с многоступенчатой настройкой (создание модели → пресет → runtime-доступ) или примеры применения для остальных
7. **Редакторные инструменты** (опционально) — кодогенерация, контекстные меню, editor-only API. Для пакетов с editor-тулами
8. **Граничные случаи** — нюансы поведения

## Если система состоит из нескольких крупных компонентов

Общий вводный блок (назначение, зависимости), затем каждый компонент как отдельный `##` раздел со своей архитектурой, API и граничными случаями. Не дублировать информацию между разделами.

## Debug-секции

Не расписывать внутреннюю механику (partial-расширения, кто от кого наследуется). Указывать:
- Что логируется автоматически
- Как получить флаг извне для профилированных логов (`Settings.Data().XxxDebugMode`)
- Как настроить (ассет `DebugSettings`, какие toggle включить)

## Что НЕ включать

- Внутренние уровни 1/2 в пользовательской документации
- Детали реализации partial-расширений (если не нужны пользователю)
- Субъективные оценки
